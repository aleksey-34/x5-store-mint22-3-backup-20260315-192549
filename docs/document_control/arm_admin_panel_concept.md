# Концепт админпанели АРМ объекта

Дата: 2026-03-11

## Цель

Собрать единое веб-АРМ для прораба/зам. прораба, где видны:

- комплектность документооборота по Форма 2
- что создано / что просрочено / что отсутствует
- журналы, наряды, задачи на день
- сканирование, печать, архивирование
- подготовка исполнительной документации

## Варианты реализации

## Вариант A (рекомендуемый): FastAPI + SSR + HTMX

Стек:

- Backend: текущий FastAPI + SQLAlchemy + SQLite
- UI: Jinja2 + HTMX + легкий JS
- Плюсы:
  - минимальные риски и быстрый запуск на текущем коде
  - нет сложной фронтенд-сборки
  - хорошо работает на слабом ноутбуке и локальной сети объекта
- Минусы:
  - менее гибкий UX, чем SPA

## Вариант B: FastAPI + React/Vite SPA

Стек:

- Backend API (уже есть)
- Frontend SPA на React
- Плюсы:
  - удобный rich-интерфейс, хорошие фильтры и таблицы
- Минусы:
  - выше стоимость поддержки и внедрения
  - больше инфраструктуры (build, deployment)

## Вариант C: Low-code слой (Appsmith/ToolJet/Directus)

Стек:

- Текущая БД + API
- Отдельный low-code UI
- Плюсы:
  - быстро сделать экран метрик и CRUD
- Минусы:
  - внешняя зависимость на платформу
  - сложнее адаптировать под полевой workflow (скан/печать/офлайн)

## Рекомендация

Для объекта и текущего этапа: Вариант A.

Причина: дает быстрый запуск MVP в локальном контуре без тяжелого фронтенда, а затем можно мигрировать в SPA без потери backend-логики.

## MVP-модули панели

1. Дашборд объекта
- % комплектности по Форма 2
- сколько приказов/журналов/нарядов заведено
- сколько документов в ручной проверке после скана
- сколько удостоверений истекает < 30 дней

2. Документооборот
- карточка каждого обязательного документа
- статусы: "нет", "черновик", "подписан", "в архиве"
- быстрый переход к файлу и печати

3. Журналы
- список обязательных журналов по "+"
- флаг "открыт/не открыт"
- напоминание о ежедневном заполнении

4. Персонал и допуски
- список сотрудников
- допуски/удостоверения и срок действия
- подсветка рисков по срокам

5. Скан-центр
- кнопка "Скан в inbox"
- таблица результатов ingest (archived/manual_review)
- быстрый просмотр OCR-текста

6. Печать
- пакетная сборка PDF
- статус "готово к печати"
- повторный экспорт без служебных колонтитулов

7. TODO на день
- авто-список задач на основе незакрытых позиций
- ручные задачи прораба
- отметки выполнения

## Интеграция с Google Docs (ЖОГР и график)

Режим: read-only синхронизация в локальное АРМ.

Что можно подтянуть:

- журнал общих работ (факт событий)
- график производства работ (план/факт)

Технически:

- подключение через Google Docs/Drive API (service account)
- периодический pull в локальную БД (каждые 15-60 минут)
- нормализация в таблицы `journal_entries` и `work_schedule`
- журнал синхронизации с версией/временем

## Риски и ограничения

- Нужны доступы к Google-документам для service account.
- Для WIA-сканирования панель должна запускаться на рабочем ноутбуке рядом с МФУ.
- Юридически значимые журналы ОТ/ПБ остаются бумажными; в системе фиксируется только статус и реестр.

## Этапы внедрения

1. Этап 1 (2-4 дня):
- дашборд + матрица комплектности + TODO

2. Этап 2 (3-5 дней):
- персонал/допуски + журналы + контроль сроков

3. Этап 3 (3-5 дней):
- скан-центр + печать + связь с исполнительной документацией

4. Этап 4 (2-4 дня):
- read-only интеграция Google Docs/Sheets

## GPU и локальная LLM на текущем ноутбуке

Фактический режим сейчас:

- Ollama запущен и работает стабильно.
- По `ollama ps` модель `llama3.2:3b` работает как `100% CPU`.
- На текущей конфигурации (Intel UHD, без дискретной GPU) практичная стратегия - CPU-first.

Рекомендация:

- Оставить `llama3.2:3b` как базовую рабочую модель.
- Для более быстрого отклика добавить fallback на более легкую модель (1-2B).
- Если появится ноутбук с NVIDIA, включить GPU-режим и использовать более емкие модели.

## Где применить ML в проекте (прикладно)

1. Классификация входящих сканов:
- автоматическое определение типа документа (приказ/удостоверение/акт/накладная)
- снижение нагрузки на ручной разбор inbox

2. Оценка риска просрочек:
- скоринг по признакам (нет журнала, истекают удостоверения, отставание по графику)
- приоритетизация задач прораба на день

3. Извлечение реквизитов из OCR:
- ФИО, номер удостоверения, срок действия, номер приказа
- автозаполнение карточек и реестров

4. Подготовка черновиков исполнительной документации:
- сводка план/факт из журнала общих работ + графика производства
- проект формулировок для актов/пояснений перед ручной проверкой

Важно: для юридически значимых документов ML используется как помощник, финальная проверка и подпись остаются за ответственным лицом.
